User Authentication P0

Handles JWT token management, login/logout flows, OAuth providers, and session persistence across all client types.

Documentation Last analyzed 12 min ago
Overview
Purpose

The User Authentication domain is the security foundation of the TaxStick platform. It manages identity verification, session tokens, and authorization gates for all API endpoints. The system supports both native email/password login and OAuth 2.0 providers (Google, Microsoft).

Architecture

Authentication follows a stateless JWT pattern. Access tokens are short-lived (15 min) and refresh tokens are stored server-side in Redis with a 7-day TTL. Token rotation is enforced on every refresh cycle to mitigate replay attacks.

Key Files
  • src/auth/login.ts — Login handler, rate limiting, brute-force protection
  • src/auth/jwt.ts — Token generation, validation, rotation logic
  • src/auth/oauth.ts — Google/Microsoft OAuth callback handlers
  • src/auth/middleware.ts — Express middleware for route protection
  • src/auth/refresh.ts — Refresh token lifecycle management
Technical Notes

Password hashing uses bcrypt with a cost factor of 12. All token payloads are unsigned (payload-visible) — no sensitive claims should be included. The sub claim carries a UUID (not the database row ID) to prevent enumeration.

Rate limiting is applied at the IP level (10 requests/min via Redis) and at the account level (5 failed logins → 15 min lockout). Unlock is automatic; no manual admin intervention is needed.

Requirements 8 functional · 4 non-functional
Functional Requirements
  • Users shall be able to register with email and password
  • Users shall be able to log in via Google OAuth 2.0
  • Sessions shall expire after 15 minutes of inactivity
  • Refresh tokens shall be rotated on every use
  • Accounts shall be locked after 5 consecutive failed logins
  • Password reset shall be initiated via email link (24h expiry)
  • All authentication events shall be logged for audit trail
  • Two-factor authentication (TOTP) shall be optionally enabled
Non-Functional Requirements
  • Login endpoint latency ≤ 200ms (p99)
  • Token validation shall be performed in < 5ms
  • Auth service shall achieve 99.9% monthly uptime
  • All credentials at rest shall be encrypted with AES-256

Bugs & Security
2 Critical 1 High ⟳ Analyzing…
CRITICAL
JWT secret falls back to hardcoded default
src/auth/jwt.ts:12

Description: The JWT signing secret is read from process.env.JWT_SECRET with a hardcoded fallback "my-secret-key". In production environments where the env var is missing, all tokens are signed with a predictable value attackers can exploit to forge arbitrary tokens.

Suggested Fix

Remove the fallback. Throw an error on startup if JWT_SECRET is absent or shorter than 32 characters.

CRITICAL
Refresh token not invalidated on logout
src/auth/login.ts:89

The logout handler clears the client-side cookie but does not delete the refresh token from Redis. An attacker who obtained the token cookie before logout can continue to refresh access tokens indefinitely.

HIGH
Timing attack in password comparison
src/auth/login.ts:44

Password comparison exits early on first mismatch, leaking timing information about partial matches. Use crypto.timingSafeEqual() instead.

Payment Processing P0

Stripe integration, invoice generation, refund workflows, and transaction audit logging.

Documentation Last analyzed 8 min ago
Overview
Purpose

The Payment Processing domain orchestrates all monetary transactions on the TaxStick platform. It integrates with Stripe as the sole payment processor, handles webhook events for async state updates, and ensures PCI-DSS compliance by never touching raw card data.

Flow

Payment Intent created server-side → client confirms with Stripe.js → webhook fires payment_intent.succeeded → invoice record created → email receipt dispatched. Retry logic (exponential backoff, 3 attempts) handles transient Stripe failures.
